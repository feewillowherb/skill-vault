## 快速开始

### 前置条件

- Git 环境
- 对多仓库项目管理的基本了解
- 可选：HagiCode 桌面应用

### 5 分钟快速上手

1. **创建主仓库结构**（见下方详细步骤）
2. **配置 `monospecs.yaml`**（见配置示例）
3. **拉取子仓库**：使用提供的 clone 脚本
4. **添加到 HagiCode**：通过桌面应用界面添加项目

## 初始化配置

### 步骤 1: 创建主仓库

```bash
# 创建项目目录
mkdir your-project
cd your-project

# 初始化 Git 仓库
git init

# 创建 git 子目录（monospecs 约定）
mkdir git
cd git
git init
cd ..
```

### 步骤 2: 配置 monospecs.yaml

在项目根目录创建 `monospecs.yaml`：

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  - path: "repos/frontend"
    url: "https://github.com/your-org/frontend.git"
    displayName: "前端应用"
    icon: "🌐"
    
  - path: "repos/backend"
    url: "https://github.com/your-org/backend.git"
    displayName: "后端服务"
    icon: "⚙️"
```

### 步骤 3: 配置 .gitignore

在项目根目录创建或更新 `.gitignore`：

```gitignore
# 排除所有子仓库内容
repos/

# 保留 monospecs 配置
!monospecs.yaml
```

### 步骤 4: 创建 clone 脚本

创建 `clone.sh`（或 `clone.bat`）：

```bash
#!/bin/bash
# 读取 monospecs.yaml 并克隆所有子仓库

# 这里需要 YAML 解析工具，如 yq
# 示例逻辑：读取 repositories 数组并执行 git clone
```

### 步骤 5: 拉取子仓库

```bash
# 执行 clone 脚本
./clone.sh

# 或手动克隆每个仓库
mkdir -p repos
cd repos
git clone https://github.com/your-org/frontend.git
git clone https://github.com/your-org/backend.git
```

### 步骤 6: 添加到 HagiCode

1. 打开 HagiCode 桌面应用
2. 点击"添加项目"或"+"按钮
3. 选择主项目文件夹
4. 验证所有子仓库被正确识别

## 日常使用

### 添加新子仓库

1. **编辑 `monospecs.yaml`**：

```yaml
repositories:
  # ... 现有仓库
  - path: "repos/new-service"
    url: "https://github.com/your-org/new-service.git"
    displayName: "新服务"
    icon: "🆕"
```

2. **拉取新仓库**：

```bash
cd repos
git clone https://github.com/your-org/new-service.git
```

3. **验证配置**：检查 HagiCode 应用中是否正确显示

### 移除子仓库

1. **编辑 `monospecs.yaml`**：删除对应的仓库条目
2. **更新 Git 忽略**：如需完全移除，手动删除 `repos/` 下的目录
3. **提交配置变更**：将 `monospecs.yaml` 的变更提交到主仓库

### 更新子仓库

```bash
# 更新所有子仓库
cd repos
for dir in */; do
  cd "$dir"
  git pull
  cd ..
done

# 或使用提供的 update 脚本
./update.sh
```

## 与 OpenSpec 集成

### 创建跨仓库变更

1. **创建变更目录**：

```bash
mkdir -p openspec/changes/add-user-auth
cd openspec/changes/add-user-auth
```

2. **创建变更文件**：

```bash
# proposal.md
# design.md  
# tasks.md
```

3. **实现代码变更**：在相应的 `repos/` 子仓库中实现功能

4. **归档变更**：OpenSpec 自动提交 specs 到主仓库

### commit_when_archive 行为

当设置 `commit_when_archive: true` 时：

- ✅ **自动提交**：specs 变更（proposal、design、tasks）自动提交到主仓库
- ❌ **不提交**：子仓库的实际代码变更需要手动提交
- 🔄 **简化流程**：省去手动管理 specs 版本控制的麻烦

### 变更流程示例

```
1. 创建变更: openspec/changes/add-user-auth/
2. 完成设计: design.md 完成
3. 实现代码: 修改 repos/frontend/src/components/Login.tsx
4. 执行归档: OpenSpec 自动提交 specs 到主仓库
5. 手动提交: git commit frontend 子仓库的代码变更
```

## 常见操作场景

### 场景 1: 多仓库协同开发

**需求**: 前端和后端独立开发，但需要统一管理

**解决方案**:

```yaml
repositories:
  - path: "repos/frontend"
    url: "https://github.com/your-org/frontend.git"
    displayName: "前端应用"
    icon: "🌐"
    
  - path: "repos/backend"  
    url: "https://github.com/your-org/backend.git"
    displayName: "后端服务"
    icon: "⚙️"
```

**操作流程**:
1. 前端开发者在 `repos/frontend/` 中工作
2. 后端开发者在 `repos/backend/` 中工作
3. 通过 OpenSpec 协调跨仓库的 API 变更
4. 主仓库追踪整体变更历史

### 场景 2: AI 辅助开发

**需求**: 为 AI Agent 提供项目结构信息

**解决方案**: 创建 `AGENTS.md` 文件

```
your-project/
├── AGENTS.md              # 主仓库规范
└── repos/
    ├── frontend/
    │   └── AGENTS.md      # 前端特定规范
    └── backend/
        └── AGENTS.md      # 后端特定规范
```

**AGENTS.md 内容示例**:

```markdown
# 项目规范

## 技术栈
- 前端: React + TypeScript + Vite
- 后端: Node.js + Express + PostgreSQL

## 代码规范
- 使用 ESLint + Prettier
- 遵循 Airbnb Style Guide
- 组件命名使用 PascalCase

## OpenSpec 集成
- 所有 API 变更需要创建 design.md
- 跨仓库变更需要在主仓库协调
```

### 场景 3: CI/CD 集成

**需求**: 批量构建和部署多个仓库

**解决方案**: 利用配置驱动的脚本

```bash
# ci-build.sh
#!/bin/bash

# 读取 monospecs.yaml
repos=$(parse_yaml monospecs.yaml)

for repo in $repos; do
  echo "Building $repo..."
  cd "repos/$repo"
  npm run build
  cd ../..
done
```

## 故障排除

### 问题 1: 子仓库未被识别

**症状**: HagiCode 应用中看不到子仓库

**解决方案**:
1. 检查 `monospecs.yaml` 语法是否正确
2. 确认路径与实际目录结构匹配
3. 验证 Git URL 是否可访问
4. 重启 HagiCode 应用

### 问题 2: 配置文件解析失败

**症状**: "Invalid YAML format" 错误

**解决方案**:
1. 使用 YAML 验证工具检查语法
2. 确保缩进使用空格而非 Tab
3. 检查特殊字符是否正确转义

### 问题 3: 子仓库代码被提交到主仓库

**症状**: 主仓库包含了子仓库的代码文件

**解决方案**:
1. 检查 `.gitignore` 是否正确配置 `repos/`
2. 清理主仓库 Git 历史（如需要）
3. 重新配置 Git 排除规则

### 问题 4: OpenSpec 归档后 specs 未提交

**症状**: 归档操作后，主仓库没有 specs 变更

**解决方案**:
1. 确认 `commit_when_archive: true` 已设置
2. 检查 OpenSpec 配置
3. 手动提交 specs 作为临时解决方案

## 最佳实践

### 开发流程

1. **主仓库优先**: 先创建主仓库和配置，再添加子仓库
2. **配置驱动**: 所有操作基于 `monospecs.yaml` 配置
3. **文档同步**: 配置变更时同步更新相关文档
4. **定期同步**: 定期更新子仓库和主仓库配置

### 团队协作

1. **规范统一**: 制定统一的命名和配置规范
2. **权限管理**: 明确主仓库和子仓库的权限
3. **代码审查**: 配置文件变更需要代码审查
4. **版本控制**: 所有配置变更纳入版本控制

### 维护建议

1. **定期检查**: 定期验证配置与实际状态一致性
2. **备份配置**: 重要配置变更前先备份
3. **监控告警**: 建立配置同步和仓库状态监控
4. **文档更新**: 及时更新使用文档和规范
