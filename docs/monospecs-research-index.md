## 核心发现总结

### Monospecs 是什么

Monospecs 是 HagiCode 提供的多仓库管理方案，通过 `monospecs.yaml` 配置文件实现：
- **统一配置管理**: 单一配置文件追踪所有子仓库
- **关注点分离**: OpenSpec 变更独立在主仓库，子仓库保持纯净
- **自动化友好**: 配置驱动的脚本和 AI Agent 支持
- **无缝集成**: 与 OpenSpec 变更管理工作流深度集成

### 核心价值主张

1. **简化多仓库管理**: 通过配置文件统一管理多个独立仓库
2. **提升开发效率**: 自动化脚本减少重复操作
3. **改善团队协作**: 结构化配置便于理解项目结构
4. **AI 友好设计**: 为 AI Agent 提供结构化项目信息

### 适用场景

**最佳适用场景**:
- 多仓库协同开发（前端、后端、桌面端独立开发）
- 跨仓库功能开发（一个功能涉及多个仓库）
- AI 辅助开发（为 AI 提供项目结构信息）
- CI/CD 集成（批量处理多个仓库）

**不太适用场景**:
- 单一仓库项目
- 仓库间无关联的独立项目
- 不使用 OpenSpec 的项目

### 关键技术特点

**配置文件结构**:
```yaml
version: "1.0"
commit_when_archive: true
repositories:
  - path: "repos/frontend"
    url: "https://github.com/org/frontend.git"
    displayName: "前端应用"
    icon: "🌐"
```

**项目结构规范**:
```
your-project/
├── openspec/          # 变更管理（独立）
├── monospecs.yaml     # 核心配置
├── repos/             # 子仓库（Git 排除）
│   ├── frontend/
│   └── backend/
└── .gitignore         # 必须排除 repos/
```

## 与普通仓库的区别

| 特性 | 普通仓库 | Monospecs 管理的项目 |
|------|---------|---------------------|
| OpenSpec 位置 | 仓库根目录 | 主仓库根目录 |
| 变更范围 | 仅限当前仓库 | 可涉及多个子仓库 |
| 子仓库纯净度 | specs 与代码混合 | specs 独立，子仓库纯净 |
| 归档提交 | 手动提交 specs | 自动提交 specs |

## 技术要点

### 核心概念

**主仓库 (Main Repo)**:
- 存储 `monospecs.yaml` 配置
- 管理 `openspec/` 变更目录
- 追踪子仓库的元数据关系

**子仓库 (Sub Repositories)**:
- 独立的 Git 仓库
- 放置在 `repos/` 目录
- 保持纯净，不包含 `openspec/`

**配置驱动**:
- 所有操作基于配置文件
- 便于工具和 AI 解析
- 支持自动化脚本

### 关键配置项

**基础配置**:
- `version`: 配置文件版本
- `commit_when_archive`: 自动提交 specs
- `repositories`: 仓库列表

**仓库配置**:
- `path`: 相对路径（必须以 `repos/` 开头）
- `url`: Git 仓库 URL
- `displayName`: 显示名称
- `icon`: 图标（emoji）

### OpenSpec 集成要点

**变更创建**:
1. 在主仓库 `openspec/` 创建变更
2. 选择目标子仓库
3. 在子仓库中实现代码变更
4. 归档时自动提交 specs

**跨仓库变更**:
- 一个变更可涉及多个子仓库
- specs 统一管理在主仓库
- 代码变更分布在各个子仓库

## 常见问题

### Monospecs 适合我的项目吗？

**适合**:
- 多个相关的独立仓库
- 需要统一管理的项目

- 需要 AI 辅助开发的项目

**不适合**:
- 单一仓库项目
- 完全独立的无关仓库
- 不使用变更管理的项目
